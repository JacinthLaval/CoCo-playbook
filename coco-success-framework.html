The Insight

The win is never the specific deliverable.
It's the process that finds it.

A governance lead says "our semantics are a mess." A CISO says "we can't pass an audit." An analytics lead says "it takes 3 weeks to get a new metric." That's the signal. Everything after that is a repeatable pattern: investigate the mess, quantify it, build a plan, establish what "before" looks like, then unleash CoCo to build the "after." The domain changes. The framework doesn't.

The pattern works because CoCo is not a single tool — it's a workbench.

Skills, search, SQL execution, file generation, deployment orchestration, Cortex Analyst, semantic views, agents — they're all available. The framework tells you when to reach for each one. Your job is to bring the business context. CoCo handles the execution surface.

Phase 1

Discover — Listen for the signal

Every successful engagement starts with someone saying something is broken. Your job is to hear the real problem behind the complaint.

🎯
Find the pain
The signal is never "we need a semantic layer." It's a symptom.

Signals to listen for

  • "Nobody agrees on the numbers"
  • "Every team has their own mart"
  • "The report says X but finance says Y"
  • "We can't trust the data"
  • "It takes 3 weeks to get a new metric"
  • "Our semantics suck" (direct hit)

What you're really identifying

  • Who owns the problem? (governance lead, CDO, analyst team lead)
  • What's the blast radius? (one team? every dashboard?)
  • Is there existing artifacts? (LookML, dbt, YAML, Excel dictionaries)
  • What's the political landscape? (who will resist, who will champion)

Real-world example — How it starts

A Director of Data Governance at a large health system said it plainly on a call: "It's a super complex mess of data with semantic layers intertwined in the complexity." That one sentence launched the engagement. The signal was governance-level frustration with multiple source systems, every team building their own mart, and metrics diverging before meetings ended.

Phase 2

Investigate — Gather the evidence

Don't guess. Get the actual artifacts. Feed them to CoCo. Let it quantify the mess so you can show the customer exactly what's broken — with numbers, not opinions.

🔍
Get the raw material
Whatever they have — LookML, YAML, SQL, CSV, Access DBs, tribal knowledge.

Common input types

  • LookML projects (views, explores, dashboards)
  • dbt models (YAML schema files, SQL models)
  • Existing semantic models / YAML files
  • Snowflake data catalog (tables, views, schemas)
  • Excel data dictionaries / Access databases
  • Confluence pages / "the doc someone wrote 3 years ago"

CoCo actions

  • Parse and extract definitions from raw files
  • Use snowflake_object_search to catalog what exists
  • Run SQL to profile tables, count rows, check freshness
  • Cross-reference definitions to find conflicts and duplicates
  • Use relevant skills (semantic-view, data-governance, data-quality)
  • Quantify: "X conflicts, Y duplicates, Z orphaned fields"

Real-world example — What investigation reveals

At a large health system, CoCo parsed dozens of LookML projects in minutes: 1,000+ table structures, tens of thousands of defined fields, hundreds of real conflicts. Duplicate explores with identical fields — zero reason for both to exist. Key metrics like Length of Stay returning completely different units across marts. Core entities like "Provider" defined multiple ways across source systems. The numbers told the story — no opinions needed.

Phase 3

Plan — Lay it out before you build

CoCo has a plan system for a reason. Use it. Break the work into phases, identify quick wins, and sequence the hard stuff after the easy wins are shipped.

📋
Structure the work
Quick wins first. Hard problems second. Always show progress early.

Plan structure

  • Prioritize by impact: what's the biggest pain?
  • Identify quick wins: what can deploy in days?
  • Sequence dependencies: what unblocks what?
  • Define "done" for each phase
  • Assign clear validation criteria
  • Plan the baseline measurement (Phase 4)

CoCo actions

  • Use CoCo plan mode to create structured plan files
  • Reference investigation findings to size each task
  • Use TODO tracking for multi-step execution
  • Identify which CoCo skills apply to each task
  • Build the plan in a format the customer can review
  • Generate the deliverable roadmap (HTML, slides, doc)

Real-world example — Structuring the plan

A typical three-phase approach: Weeks 1-4 — deploy semantic views for the cleanest schemas first, merge duplicates, get Cortex Analyst live on the migrated layer. Months 2-3 — migrate remaining projects, resolve conflicting field definitions, unify core entities across sources. Quarter 3 — build toward MDM on a clean semantic foundation. Quick wins (fields already identical across marts) ship immediately. Conflicts get sequenced by schema — start where the highest concentration lives.

Phase 4

Baseline — Establish the "before"

You cannot prove value without a before. Measure the current state in a way that's concrete, undeniable, and easy to compare against later.

📊
Measure what matters
Pick metrics that the customer cares about, not metrics that make CoCo look good.

Baseline metrics by use case

  • Semantic/Governance: # of conflicting definitions, # of orphaned fields, time to answer a question
  • Analytics: time to build a new metric, # of ticket requests in queue, dashboard trust score
  • Data quality: # of DMF violations, % of tables monitored, freshness SLA misses
  • AI/ML: model accuracy, inference latency, time from training to deployment
  • Migration: # of objects to migrate, % converted, test pass rate

CoCo actions

  • Run the same queries the customer would run — capture the output
  • Screenshot / export the current state of dashboards
  • Use data-quality skill to audit current DMF coverage
  • Use data-governance skill to snapshot access patterns
  • Generate a baseline report (HTML, Google Doc, or Snowflake table)
  • Store baseline in Snowflake for automated comparison later

Real-world example — Capturing the baseline

Hundreds of conflicts across tens of thousands of fields. Duplicate objects with identical definitions that nobody knew about. Key clinical metrics returning different units depending on which mart you queried. Core entities defined multiple ways across source systems. The majority of conflicts concentrated in one project family. These numbers became the "before" that every subsequent milestone measured against.

Phase 5

Execute — Unleash CoCo

Now you build. Use CoCo's skills, tools, and deployment capabilities to ship the solution. Show the delta against baseline at every checkpoint.

🚀
Ship, measure, iterate
Deploy quick wins fast. Build momentum. Tackle hard problems with confidence.

Execution principles

  • Ship the quick wins first — build trust and momentum
  • Use CoCo skills for domain-specific work (semantic-view, dbt, streamlit, etc.)
  • Test everything before declaring victory
  • Show delta against baseline at every checkpoint
  • Document as you go — CoCo memory persists context
  • Let the customer validate: "do these numbers match?"

CoCo capabilities

  • semantic-view — create, debug, evaluate semantic views
  • sql-author — write, test, deploy SQL
  • data-quality — attach DMFs, monitor, alert
  • cortex-agent — build NL interfaces for analysts
  • dashboard — create visual proof of value
  • developing-with-streamlit — build interactive apps

Real-world example — Shipping the result

Thousands of semantic views extracted from legacy BI — across every schema, zero code written. Hundreds of views ready to deploy immediately across the cleanest domains. Identical fields across marts deployed same day — no meeting needed. The live validation test: pick a dashboard the governance team uses today, deploy the semantic views, point Cortex Analyst at the same data, ask the same questions in plain English. If the numbers match, the migration is proven.

Beyond Semantic Layers

The framework applies to any CoCo engagement

The domain changes. The five phases don't. Here's how the same pattern maps to different starting signals.

Governance

"We don't know who has access to what"

Discover: CISO flags audit risk. Investigate: CoCo runs data-governance skill, snapshots roles/grants. Plan: prioritize sensitive schemas. Baseline: current grant sprawl. Execute: masking policies, row access, classification.

Clinical AI

"We need to deploy a model on our data"

Discover: clinician has a research model. Investigate: CoCo profiles the data, checks GPU infra. Plan: SPCS service + Streamlit UI + audit trail. Baseline: manual process time. Execute: containerized inference, live dashboards.

Migration

"We're moving off Teradata/Oracle"

Discover: CFO wants to reduce license spend. Investigate: CoCo parses DDL, maps dependencies. Plan: table-by-table migration waves. Baseline: object count + query compatibility. Execute: SnowConvert + validation testing.

Analytics

"Our analysts are drowning in SQL tickets"

Discover: analytics lead says turnaround is 2 weeks. Investigate: audit JIRA queue, profile common queries. Plan: semantic views + Cortex Analyst. Baseline: avg ticket resolution time. Execute: NL query interface, self-service.

Data Pipelines

"Our pipelines break every Monday"

Discover: data eng is firefighting. Investigate: CoCo audits dynamic tables, checks refresh history. Plan: DMF monitoring + alerting. Baseline: failure count per week. Execute: quality monitors, circuit breakers, SLA dashboards.

Cortex Agents

"We want an AI assistant for our team"

Discover: team lead wants NL queries. Investigate: CoCo catalogs available tables, checks semantic coverage. Plan: agent + search service + semantic views. Baseline: questions that couldn't be answered. Execute: agent + Streamlit chat UI.

Common Pitfalls

What separates great from mediocre

Skip the investigation

  • Jump straight to building without understanding the mess
  • Assume you know the schema from a 5-minute glance
  • Don't quantify the problem — just describe it
  • Ignore existing artifacts (LookML, dbt, docs)

Earn the right to build

  • Parse every artifact the customer gives you
  • Show them numbers they didn't know about their own data
  • Let the evidence dictate the plan, not assumptions
  • Use CoCo skills to automate the discovery

No baseline, no proof

  • Build something impressive but can't show the delta
  • "It's better now" without numbers to back it up
  • Customer can't tell stakeholders what changed
  • No before/after to justify the next phase

Measure the before, celebrate the after

  • Capture concrete baseline metrics before building
  • "Hundreds of conflicts → 0" is a story that sells itself
  • Every checkpoint shows progress against baseline
  • Customer presents the delta to their leadership

Boil the ocean

  • Try to fix everything at once
  • 6-month plan with no deliverables until the end
  • Customer loses interest before seeing value
  • Scope creep kills momentum

Quick wins build momentum

  • Ship the easy stuff in days, not weeks
  • Identical fields deploy today — no meeting needed
  • Each win earns trust for the harder work ahead
  • Sequence: easy → medium → hard → strategic
The Takeaway

CoCo doesn't replace your expertise.
It amplifies your process.

The framework is always the same: hear the signal, investigate the evidence, plan the work, measure the before, and ship the after. CoCo accelerates every phase — but the human judgment of which problem to solve and what "good" looks like is yours.
